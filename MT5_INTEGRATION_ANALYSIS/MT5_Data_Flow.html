📥 insert_data.php - Account Balance Data

POST

URL: {BRIDGE_DOMAIN}/insert_data.php

Purpose: Receive and store MT5 live account balance, equity, and trade metrics

📋 Input Fields:

{ "mt5_id": "413536535", // Account identifier "client_name": "Omer Qureshi", // Trader name "daily_drawdown_limit": 8, // Max daily loss (%) "overall_drawdown_limit": 12, // Max overall loss (%) "weekly_profit_target": 10, // Profit target (%) "running_balance": 10013.31, // Current balance ($) "running_equity": 10013.31, // Current equity ($) "largest_single_trade_profit_week": 0, // Top trade profit ($) "trades_under_60s_week": 0 // Count of 60s+ trades }

⚙️ Processing Flow:

1. Receive POST Data Read raw data from php://input
2. Clean & Validate Remove null chars, parse JSON, check fields
3. Save Locally Write to received_data.json for debugging
4. Send to Supabase POST to account_data_extended table (upsert)
5. Log Results Write all operations to debug_log.txt

🗄️ Database Operation:

Supabase REST API: POST {SUPABASE_URL}/rest/v1/account_data_extended?on_conflict=mt5_id Headers: - Content-Type: application/json - apikey: [SUPABASE_ANON_KEY] - Authorization: Bearer [SUPABASE_ANON_KEY] - Prefer: resolution=merge-duplicates,return=minimal Behavior: UPSERT - Updates if mt5_id exists, inserts if new

⚠️ update_trading_account.php - Trading Account Status

PATCH

URL: {BRIDGE_DOMAIN}/update_trading_account.php

Purpose: Update trading account status and breach alerts (rule violations)

📋 Input Fields:

{ "mt5_login": "435304224", // Account login (required) "status": "breached", // Status: active, breached, suspended "breach_reason": "daily limit hit" // Reason for breach }

⚙️ Processing Flow:

1. Receive & Validate Read POST data, validate mt5_login presence
2. Save Locally Write to received_trading_account.json
3. Send PATCH to Supabase Update trading_accounts table where mt5_login matches
4. Log Response Log HTTP status and response to debug_log_update.txt

🗄️ Database Operation:

Supabase REST API: PATCH {SUPABASE_URL}/rest/v1/trading_accounts?mt5_login=eq.[LOGIN] Headers: - Content-Type: application/json - apikey: [SUPABASE_ANON_KEY] - Authorization: Bearer [SUPABASE_ANON_KEY] - Prefer: resolution=merge-duplicates,return=representation Body: Updated status and breach_reason fields only

🔒 Security Analysis

🚨 CRITICAL: API Keys Exposed in Debug Logs

Supabase API keys are hardcoded in PHP files and logged in plaintext to debug_log.txt. This allows unauthorized database access.

⚠️ HIGH: Database Credentials in Plaintext

MySQL credentials are hardcoded in PHP files. Should use environment variables instead.

⚠️ HIGH: No Authentication on Endpoints

insert_data.php and update_trading_account.php have no validation that requests come from legitimate MT5 systems. Anyone can call these endpoints.

⚠️ MEDIUM: Sensitive Data in JSON Files

Client names, MT5 IDs, and balances stored in received_data.json in web root. Could be accessed by unauthorized users.

⚠️ MEDIUM: Debug Mode Enabled

error_reporting(E_ALL) and display_errors enabled in production code. Could expose sensitive information in error messages.

✅ Recommendations:

  • Move secrets to environment variables (.env)
  • Implement API key authentication on endpoints
  • Use HTTPS only, implement request signing
  • Disable debug logging in production
  • Store sensitive data in database, not JSON files
  • Implement rate limiting to prevent abuse
  • Add input validation and sanitization
  • Log audit trails for all data modifications
